Java 8 (2014) — What's New, Part 1

The upgrade that made Java functional, expressive & powerful

Java 8 is the single most important release for interviews. Part 1 covers the functional core: lambdas, functional interfaces, streams, method references and default methods.

1

Lambda Expressions

Pass behavior as data — write a function inline, without a class or an anonymous type.

Syntax
  • (parameters) -> expression
  • (parameters) -> { statements; }
  • No parameters: () -> doWork()
Example 
// Before Java 8 — anonymous class noise
Runnable r1 = new Runnable() {
    public void run() { System.out.println("Hello"); }
};

// Java 8 — one line
Runnable r2 = () -> System.out.println("Hello");

Comparator<User> byAge = (a, b) -> a.getAge() - b.getAge();
Less boilerplate. Behavior becomes a value you can pass around.
2

Functional Interfaces

Interfaces with exactly one abstract method — the target type every lambda needs.

The Big Four
  • Predicate<T> — takes T, returns boolean → test something
  • Function<T,R> — takes T, returns R → transform something
  • Consumer<T> — takes T, returns nothing → act on something
  • Supplier<T> — takes nothing, returns T → provide something
Example 
// 1. Predicate — test a condition, returns boolean
Predicate<String> isLong = s -> s.length() > 5;
isLong.test("Hello");          // false
isLong.test("Hello Java");     // true

// 2. Function — transform T into R
Function<String, Integer> length = s -> s.length();
length.apply("Java");          // 4

// 3. Consumer — accept a value, return nothing
Consumer<String> print = s -> System.out.println(s);
print.accept("Hello");         // prints: Hello

// 4. Supplier — provide a value, take nothing
Supplier<Double> random = () -> Math.random();
random.get();                  // e.g. 0.7431
Interview favourite: know all four signatures cold — and BiFunction, UnaryOperator as bonus.
3

Streams API

Declarative data processing: describe what you want, not how to loop for it.

Pipeline Anatomy
Source
list.stream()→ Intermediate ops
filter · map · sorted · distinct→ Terminal op
collect · forEach · reduce · count
Example 
// Before Java 8 — loop + if + temp list + sort
List<String> result = new ArrayList<>();
for (User u : users) {
    if (u.isActive()) {
        result.add(u.getName());
    }
}
Collections.sort(result);

// Java 8 — the same logic as one readable pipeline
List<String> result = users.stream()
    .filter(User::isActive)      // keep active users
    .map(User::getName)          // User → String
    .sorted()                    // alphabetical
    .collect(Collectors.toList());
  • Lazy — nothing runs until the terminal operation
  • One-time use — a consumed stream cannot be reused
  • Doesn't modify the source collection
Streams process data; Collections store data. Say this sentence in every interview.
4

Method References

A shorthand for lambdas that only call an existing method. Cleaner, and reads like English.

Four Kinds
  • Static method → Integer::parseInt
  • Instance method of an object → System.out::println
  • Instance method of a type → String::toUpperCase
  • Constructor → ArrayList::new
Example 
// Lambda (before)                    Method reference (after)
// 1. Static method
s -> Integer.parseInt(s)       →   Integer::parseInt
// 2. Instance method of a specific object
s -> System.out.println(s)     →   System.out::println
// 3. Instance method of an arbitrary object of a type
s -> s.toUpperCase()           →   String::toUpperCase
// 4. Constructor
() -> new ArrayList<>()        →   ArrayList::new
// In real code — both lines do exactly the same:
names.sort((a, b) -> a.compareToIgnoreCase(b));  // lambda
names.sort(String::compareToIgnoreCase);         // method ref ✔
If a lambda's body just calls one method, replace it with a method reference.
5

Default & Static Methods in Interfaces

Interfaces can now carry implementations — the trick that let Java add stream() to every Collection without breaking the world.

Example 
interface Vehicle {
    String name();                       // abstract — must implement
    default String honk() {              // default — comes for free
        return name() + " says beep!";
    }
}

class Car implements Vehicle {
    public String name() { return "Car"; }
    // honk() is inherited — no need to write it
}

new Car().honk();   // "Car says beep!"
  • Evolve interfaces without breaking existing implementations
  • Diamond problem? The class must override and pick: Vehicle.super.honk()
Interview trap: what happens when two interfaces define the same default method? You must override.

Java 8 (2014) — What's New, Part 2

Optional · Date & Time · CompletableFuture · Parallel Streams · Nashorn

Part 2 covers the safety and concurrency side of Java 8 — the features that killed NullPointerException surprises and made async programming humane.

6

Optional

Handle values that might be missing — clean & safe. Makes "can be absent" part of the type instead of a hidden null.

Useful Methods
  • of(), ofNullable(), empty()
  • isPresent(), isEmpty() (11+)
  • ifPresent(), ifPresentOrElse()
  • orElse(), orElseGet(), orElseThrow()
  • map(), flatMap(), filter()
Example 
// Before Java 8 — defensive null checks everywhere
User user = findUser(id);
String name = "Guest";
if (user != null && user.getName() != null) {
    name = user.getName();
}

// Java 8 — absence is explicit, handled in one chain
String name = findUser(id)        // returns Optional<User>
    .map(User::getName)           // runs only if present
    .orElse("Guest");             // fallback if empty

// Run something only when a value exists
findUser(id).ifPresent(u -> System.out.println(u));
Cleaner code. No more NullPointerException!
7

Date & Time API

Modern, immutable and thread-safe date & time handling in java.time. Goodbye old Date & Calendar APIs!

Key Classes
  • LocalDate — date only · LocalTime — time only · LocalDateTime — both
  • Instant — a machine timestamp (epoch)
  • Duration — time-based amount · Period — date-based amount
  • ZonedDateTime — with time zone · DateTimeFormatter — parsing & formatting
Quick Example 
// Before Java 8 — mutable & confusing (month is 0-based!)
Date d = new Date();
Calendar cal = Calendar.getInstance();
cal.set(2026, 6, 3);           // 6 means JULY ?!
cal.add(Calendar.DAY_OF_MONTH, 7);

// Java 8 — immutable, readable, thread-safe
LocalDate today     = LocalDate.now();          // 2026-07-03
LocalDate nextWeek  = today.plusDays(7);        // 2026-07-10
LocalDate lastMonth = today.minusMonths(1);     // 2026-06-03

DateTimeFormatter fmt =
    DateTimeFormatter.ofPattern("dd-MM-yyyy");
String date = today.format(fmt);                // "03-07-2026"
Every method returns a new object — that's what makes it thread-safe.
8

CompletableFuture

Simplifies async programming with non-blocking, chainable operations. Designed for scalability and performance.

Common Methods
  • supplyAsync(), thenApply() — start & transform
  • thenAccept(), thenRun() — consume the result
  • thenCombine(), thenCompose() — join & chain futures
  • exceptionally(), whenComplete() — handle errors
Flow at a Glance
▶ Start
async task→ ⚙ Process
result→ ✔ Handle / save
result→ ✖ Handle
error 
// Before — Future can only block and wait
Future<User> future = executor.submit(() -> fetchUser(id));
User user = future.get();      // ⛔ thread stuck here until done

// Java 8 — non-blocking chain, each step runs when ready
CompletableFuture.supplyAsync(() -> fetchUser(id))  // start async
    .thenApply(User::getEmail)                      // transform
    .thenAccept(mailer::send)                       // consume
    .exceptionally(ex -> {                          // recover
        log.error(ex);
        return null;
    });
Future can only block-and-get. CompletableFuture composes — that's the interview answer.
9

Parallel Streams

Process large datasets faster using multiple cores. Leverage all cores — use wisely!

Example 
// Sequential — runs on one core
long total = employees.stream()
    .filter(Employee::isActive)
    .mapToLong(Employee::getSalary)
    .sum();

// Parallel — same pipeline, all cores. One word changed.
long total = employees.parallelStream()
    .filter(Employee::isActive)
    .mapToLong(Employee::getSalary)
    .sum();
Keep in Mind
  • Not always faster than sequential streams — splitting has overhead
  • Avoid for I/O-bound operations — it uses the common ForkJoinPool
  • Beware of shared mutable state — it breaks parallel safety
Best for large, CPU-bound, stateless operations. Measure before you parallelize.
10

Nashorn Engine

Run JavaScript inside the JVM — for scripting and integrations.

Example 
ScriptEngine engine = new ScriptEngineManager()
    .getEngineByName("nashorn");
Object result = engine.eval("2 + 3 * 4");
System.out.println(result); // 14
Note
  • Nashorn was deprecated in Java 11
  • Removed in Java 15 — know this for trivia questions
Historical footnote today — mention deprecation and you sound senior.
Java 8 — The upgrade that made Java modern. Functional. Expressive. Powerful.

Java 17 (2021) LTS

Powerful upgrades for modern applications

The enterprise standard. Spring Boot 3 requires Java 17 — this is the version most production backends run today. Six upgrades define it.

1

Sealed Classes

More control over inheritance. Define exactly who can extend or implement.

Example 
sealed class Shape permits Circle, Square {}
final class Circle extends Shape {}
final class Square extends Shape {}
Stronger safety. Easier maintenance. Enables exhaustive switch over subtypes.
2

Pattern Matching for instanceof

Type checks made simple and safe.

Example 
// Before — check, then cast manually
if (obj instanceof String) {
    String s = (String) obj;       // repetitive cast
    System.out.println(s.length());
}

// Java 17 — check + bind in one step
if (obj instanceof String s) {
    System.out.println(s.length()); // s ready to use
}
No need for explicit casts — check and bind in one step.
3

Strong Encapsulation

Better control over internal APIs using the module system — JDK internals are fully hidden by default.

Example 
module com.example.app {
    requires java.base;
    exports com.example.api;
    // internal packages are hidden
}
More security. Cleaner APIs. Libraries can no longer poke at sun.misc internals.
4

Better GC

ZGC improvements for low latency and high throughput.

Benefit
  • Lower pause times — sub-millisecond even on big heaps
  • Handles large heaps efficiently
  • Ideal for high-scale applications
Dashboard-style workloads with big result sets love ZGC's short pauses.

The modern syntax toolkit born in 12–16 is production-grade in 17. This LTS bundles it all.

? Frequently Asked Interview Questions — tap to reveal the answer
Stream vs Collection?
A Collection stores data in memory and can be traversed many times. A Stream processes data — a lazy, one-time pipeline that stores nothing and doesn't modify its source.
map() vs flatMap()?
map() transforms each element one-to-one (Stream<T> → Stream<R>). flatMap() maps each element to a stream and flattens the results — use when each element yields multiple values, like List<List<T>> → Stream<T>.
reduce() vs collect()?
reduce() folds elements into a single immutable value (sum, min, max). collect() performs mutable accumulation into a container (List, Map, StringBuilder) and is designed to be safe for parallel streams.
Optional.of() vs ofNullable()?
of() throws NullPointerException if given null — use when null would be a bug. ofNullable() returns Optional.empty() for null — use when null is a legal, expected state.
Why was Optional introduced?
To make absence explicit in the type system. Instead of a hidden null waiting to explode, the API says "this may be empty" and forces the caller to decide — eliminating a whole family of NullPointerExceptions.
Predicate vs Function vs Consumer vs Supplier?
Predicate<T>: T → boolean (test). Function<T,R>: T → R (transform). Consumer<T>: T → void (act). Supplier<T>: () → T (provide).
CompletableFuture vs Future?
Future can only block with get(). CompletableFuture is composable and non-blocking: chain with thenApply/thenCompose, merge with thenCombine, recover with exceptionally — and complete it manually.
Platform Threads vs Virtual Threads?
A platform thread wraps an expensive OS thread — you get a few thousand. A virtual thread is a cheap JVM-managed thread multiplexed onto carrier threads — you can run millions, perfect for I/O-heavy services.
Why are Virtual Threads scalable?
Because blocking becomes nearly free: when a virtual thread blocks on I/O, the JVM unmounts it and reuses the OS carrier thread for other work. Thread-per-request scales again — no reactive gymnastics needed.
Record vs Class?
A record is a concise, immutable data carrier — canonical constructor, accessors, equals, hashCode and toString all generated. A class is for mutable state, inheritance and behavior. Records can't extend classes and their fields are final.
Sealed Class vs Final Class?
final forbids all inheritance. sealed allows it — but only for an explicit permits list. You keep polymorphism and control, enabling exhaustive switch over the known subtypes.
var limitations?
Local variables only — never fields, method parameters or return types. Must be initialized at declaration, can't be initialized with null alone, no array initializer shorthand. It's still static typing — the compiler fixes the type once.
